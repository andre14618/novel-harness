### 6. Overnight Claude loop operating rules

These items are the execution contract for long unattended runs. The goal is evidence accumulation, not prompt churn.

- [ ] **Create a tracked-work row before each overnight loop.** Use `harness.experiments.createTuningExperiment(type, ...)` with `type='ticket'` for engineering loops and a specific experiment type for eval/probe loops. Every result must conclude the experiment and link to the git commit or persisted eval row.
- [ ] **Create or update a durable loop context before running unattended work.** Write a short tracked context file under `docs/sessions/` or a loop-specific result doc before the loop starts. It must include objective, starting commit, experiment ID, files/scripts expected to change, budget cap, command plan, stop condition, and pickup instructions. Do not rely on chat history as the only context.
- [ ] **Use the approved DeepSeek V4 Flash budget deliberately.** Treat `$26` as the hard overnight cap across all DeepSeek-heavy methodology probes. Default per-loop cap: `$4` unless the loop has an explicit reason to exceed it. Prefer cache-friendly stable prompts, repeated volatile-tail calls, fixed panels, paired replay, and concurrent same-lane reruns over rewriting large prefixes.
- [ ] **Use concurrency for statistical power inside one lane.** Before launching concurrent DeepSeek V4 Flash calls, declare sample shape, N, family key or panel identity, expected cost, budget cap, and promotion/rejection gate. Do not use concurrency to validate unrelated runtime lanes in parallel.
- [ ] **One primary lane per loop.** Each loop must state: goal, baseline, changed runtime lever, feedback signal, stop gate, and escalation rule before running. Do not combine prompt edits, routing changes, schema changes, checker threshold changes, planner/context changes, and retry-policy changes unless they are one declared behavior bundle.
- [ ] **Keep support work separate from runtime evidence.** Tests, replay/fixed-panel harnesses, docs-impact audits, operator summaries, stop classifiers, and result docs may run in parallel, but they do not count as evidence that the primary runtime lever worked.
- [ ] **Persist every probe that informs directionality.** Use `phase_eval_runs`, checker result JSONL, DB experiment results, or a tracked result doc. No overnight finding may live only in stdout, a terminal scrollback, or chat.
- [ ] **Commit every coherent completed loop unit.** Code, prompt, script, and durable-doc changes must be committed after relevant checks pass. The commit message must include the experiment ID or result row when applicable. Do not batch unrelated loop outcomes into one commit; do not leave successful overnight work only in an uncommitted worktree.
- [ ] **Run docs-impact discipline before every runtime commit.** For any loop touching runtime surfaces (`src/agents/**`, `src/phases/**`, `src/lint/**`, `src/llm.ts`, `src/transport.ts`, `src/models/**`, `src/config/pipeline.ts`, `sql/**`), run `bun scripts/preflight-docs-impact.ts --strict` before commit. The commit must include `docs/current-state.md` or a justified `docs-impact: none` footer; validation/result docs alone do not satisfy current-truth discipline.
- [ ] **Keep failed/aborted loop state recoverable.** If a loop stops midstream, update the durable context with last command, last log path, observed failure, remaining budget, and next safe command. Commit the context update if it changes tracked docs and no secrets/runtime artifacts are included.
- [ ] **Record negative results as first-class outcomes.** If an intervention regresses, revert or park it, append the lesson to `docs/lessons-learned.md`, and close the todo item with the evidence ID. Do not leave failed prompt edits in variant files because they are “interesting.”
- [ ] **Prefer paired comparisons with shared upstream artifacts.** When testing prompts or checker policies, keep concept/plotter/skeleton inputs shared wherever possible so the measured delta belongs to the lever under test.
- [ ] **Require promotion-grade evidence before changing runtime defaults.** Single n=10 planner probes are directional only. Runtime prompt/checker promotions need either multi-run confirmation, multi-seed confirmation, or a calibrated fixed panel with recall/precision deltas.
- [ ] **Keep checker severity conservative.** Hallucination and functional-state semantic findings stay warning-class unless calibrated precision/recall and deterministic guardrails justify blocker behavior. Adherence-events may become stricter only through a measured two-stage design.
- [ ] **Update docs in the same session and commit the update.** Any loop that changes current runtime behavior must update `docs/current-state.md` in the same commit or explicitly mark `docs-impact: none`; any loop that changes methodology must update `docs/decisions.md` or `docs/lessons-learned.md`; completed pending items must be removed or closed in this file. Commit docs separately when practical if code and documentation represent distinct concerns, but do not split runtime-current-state updates away from the runtime behavior commit.
- [ ] **Deploy before LXC evidence runs when runtime behavior changes.** Commit first, deploy via `scripts/deploy-lxc.sh`, confirm no active generation, then run LXC experiments. Do not use local-only runtime changes as evidence for deployed behavior.
