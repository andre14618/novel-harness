## What Was Learned

- **Continuity blockers are now the dominant remaining halt class for fantasy-debt.** With L17/L22/L24 all closed, the next bottleneck for unattended runs is checker classes that bypass chapter-attempt retry. The `drafting.ts:1116-1133` site has `buildCheckerBlockerDeviations` collecting continuity + accepted beat blockers + functional blockers — ANY non-empty result halts immediately. Worth a design conversation about which checker classes should be retry-eligible vs. operator-escalation-only.
- **L31d's continuity fire is a real state-divergence catch.** This is the checker doing its job. The question is whether the chapter-plan-reviser (or a continuity-aware beat rewriter) could have closed it without operator intervention.

## Lessons for `docs/lessons-learned.md`

Two candidate generalizable lessons (both derivable from this session):

1. **"When a checker class never gets a chapter-attempt retry, it becomes the dominant halt class as soon as upstream classes are fixed." (Generalized)** — When closing a cluster of failures, audit whether the next-most-frequent failure class has access to the same retry budget. If not, fixing the dominant class will surface the next class one-for-one without affecting throughput.
2. **"Multi-attempt acceptance criteria should distinguish 'attempts the system actually got to use' from 'attempts the system would have used'."** — Chapter 2 was set up for `attempts 1/3`, but the continuity blocker triggered immediate plan-assist gate routing — attempt 2/3 was never reached. An operator looking at the attempt counter alone would think the chapter "hadn't tried hard enough" when in fact the bypass behavior is by design.
