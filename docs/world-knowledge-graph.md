# World Knowledge Graph & Scoped Context Assembly

The world knowledge graph gives the novel harness structured worldbuilding that scales to 20-30 chapter novels. Instead of a flat world bible JSON blob, the system maintains typed entities (systems, cultures, relationships, timeline events, character knowledge) that evolve chapter-by-chapter and are assembled into POV-scoped context for the writer agent.

## Data Model

All tables live in the central Postgres DB (`novel_harness_orchestrator`), partitioned by `novel_id`. Schema in `sql/010_novel_data.sql` and `sql/011_vector_graph.sql`.

### World Systems (`world_systems`)

Structured representations of magic, religion, politics, economy, technology, or social systems.

| Field | Description |
|-------|-------------|
| `name` | System name (e.g., "Allomancy", "The Merchant Guild") |
| `type` | magic, religion, politics, economy, technology, social |
| `description` | How the system works — its mechanics |
| `rules_json` | Specific rules and constraints (array) |
| `manifestations_json` | How ordinary people encounter this system in daily life |
| `vocabulary_json` | Terms, jargon, slang associated with this system |
| `constraints_json` | What the system CANNOT do — its limits and costs |

Generated by the world-builder agent during concept phase. For contemporary/realistic genres, "systems" may be social structures (class hierarchies, legal systems, corporate power).

### Cultures (`cultures`)

First-class entities representing distinct cultural groups in the world.

| Field | Description |
|-------|-------------|
| `name` | Culture name |
| `description` | Who these people are and how they live |
| `values_json` | Core beliefs and values |
| `taboos_json` | Forbidden behaviors and topics |
| `speech_influences` | How culture shapes speech (formality, expressions, avoided topics) |
| `customs_json` | Daily-life practices and rituals |
| `system_views_json` | How this culture views each world system (`{systemId: perspective}`) |

Different cultures perceive the same world systems differently — a ruling class celebrates a political system the underclass resents.

### Character ↔ Culture (`character_cultures`)

Maps characters to their cultures with a relationship type:

- `native` — born into, speaks/acts as natural member
- `adopted` — joined later, may code-switch
- `outsider` — interacts with but doesn't belong to
- `rebel` — born into but actively rejects
- `exile` — forced out, carries marks but not welcome

### Character System Awareness (`character_system_awareness`)

Per-character awareness level of each world system. Evolves as characters learn:

| Level | What the writer sees | Example |
|-------|---------------------|---------|
| `ignorant` | System omitted from context entirely | Peasant doesn't know magic exists |
| `rumors` | Vague description, no mechanics | "People whisper about strange powers" |
| `aware` | Name + basic description + public rules | Knows Allomancy exists, knows metals are involved |
| `practitioner` | Full rules + vocabulary + manifestations | Burns metals, knows the sensations |
| `expert` | Everything including limitations and edge cases | Knows Allomancy's theoretical limits |

### Evolving Relationships (`relationship_states`)

Relationship between character pairs, versioned by chapter number.

| Field | Description |
|-------|-------------|
| `character_a`, `character_b` | The pair |
| `chapter_number` | When this state was recorded |
| `trust_level` | deep_trust, trust, cautious, neutral, wary, suspicious, hostile |
| `dynamic` | Current relationship description |
| `tension` | Active source of tension |
| `recent_shift` | What changed and why |

The assembler always reads the latest state before the current chapter. Static relationships from character profiles serve as baseline; these evolve on top.

### Timeline Events (`timeline_events`)

Chronological event log with participant and witness tracking.

| Field | Description |
|-------|-------------|
| `event` | WHO did WHAT with WHAT CONSEQUENCE |
| `location` | Where it happened |
| `participants_json` | Characters who actively participated |
| `witnesses_json` | Characters who saw/heard but didn't participate |
| `consequences` | What this event changed or set in motion |

Enables two queries the flat facts table can't: "what happened at this location before?" and "what events has this character witnessed recently?"

### Character Knowledge (`character_knowledge`)

Structured tracking of what each character knows, replacing flat `knows[]`/`doesNotKnow[]` strings.

| Field | Description |
|-------|-------------|
| `character_id` | Who knows this |
| `knowledge` | The specific thing they know |
| `source` | How learned: witnessed, told, overheard, deduced, read, discovered |
| `chapter_learned` | When they learned it |
| `category` | event, secret, relationship, system, location, identity |
| `is_false` | Character believes something incorrect (dramatic irony) |

## Extraction Pipeline

After each chapter is approved, five steps run:

1. **4 extractors in parallel:**
   - **summary-extractor** — chapter summary, key events, emotional state, open threads
   - **fact-extractor** — 20-35 categorized facts
   - **character-state** — location, emotional state, knows/doesNotKnow
   - **relationship-timeline** — relationship changes, timeline events, knowledge gains, awareness changes

2. **Embedding** — batch embed all new chapter data via `text-embedding-3-large` (1536 dims). Stored in vector columns on each table. Cost: ~$0.0003/chapter.

3. **Graph linker** (`src/agents/graph-linker/`) — identifies causal chains and knowledge propagation across the extracted data. Writes to `event_causes`, `knowledge_propagation`.

### Graph Tables (sql/011_vector_graph.sql)

| Table | Purpose |
|-------|---------|
| `event_causes` | Cause → effect links between timeline events (causes, enables, prevents, motivates) |
| `knowledge_propagation` | Who transmitted knowledge to whom, via what event, with confidence |
| `retrieval_config` | Per-novel tunable retrieval parameters (12 fields) |

## Context Assembly

The writer context builder (`src/agents/writer/context.ts`) uses a fixed skeleton with dynamic semantic retrieval.

### Fixed Skeleton (always included)
- **Scene Setup** — chapter number, POV character, setting, beats
- **POV World View** — cultural background, values, taboos, speech influences
- **Character Profiles** — present characters with relationship arcs (full trajectory, not just latest snapshot)
- **Craft Reminders** — POV speech pattern, voice contrast, show-don't-tell

### Dynamic Sections (semantic retrieval via `src/db/retrieval.ts`)

Scene query embedded → 6-table hybrid RRF search → character/location boost → recency decay.

Assembled based on what's actually found and relevant to the scene:
- **World Systems** — only if semantically relevant (a kitchen scene won't get magic details)
- **Setting & Atmosphere** — location details, what happened here before
- **Story Context** — relevant summaries from any chapter, not just last 5
- **Causal Context** — event chains traced via `event_causes` graph
- **Knowledge Context** — POV character's knowledge with propagation sources and confidence
- **Established Facts** — semantically relevant facts from any chapter

### Retrieval Parameters (tunable via UI or daemon)

Stored in `retrieval_config` per novel, adjustable at `/app/context`:

| Parameter | Default | Purpose |
|-----------|---------|---------|
| `max_facts` | 40 | Facts returned per scene |
| `max_events` | 15 | Timeline events returned |
| `max_summaries` | 8 | Chapter summaries returned |
| `min_similarity` | 0.25 | Cosine similarity floor |
| `rrf_k` | 60 | RRF fusion constant |
| `character_boost` | 2.0 | Score multiplier for present characters |
| `location_boost` | 1.5 | Score multiplier for location matches |
| `recency_half_life` | 10 | Chapters until recency bonus halves |

## Concept Phase Ordering

The concept phase runs agents in this order:

1. **World-builder** (first, alone) — produces world bible with systems and cultures
2. **Character-agent** + **Plotter** (parallel) — character-agent receives the world bible to assign cultural membership and system awareness per character

This adds ~5-10s wall-clock time but eliminates the need for a reconciliation step between characters and world.
